<h2 id="01">1. はじめに</h2>
<p>データベース接続の失敗やファイルの書き込みの失敗など、プログラム実行中にエラー (例外) が発生した場合、.NET 対応の Visual Basic では構造化例外処理と呼ばれる構文を使用して、例外に対処するコードを記述することができます。</p>
<p>この構造化例外処理では、Try ... Catch ... Finally 文を使用して、Try ブロック、Catch ブロック、および Finally ブロックという 3 つのブロックを構成しています (具体例は後述します)。この際、Catch ブロックが実際の例外に対処するコードを記述するため、Finally ブロックは「おまけ」のような印象があるかも知れません。実際、Finally ブロックを書かなくても、Catch ブロックだけで例外に対処できますし、文法としても Finally ブロックの省略は可能です。しかし、Finally
 ブロックも Catch ブロックに優るとも劣らない重要な役割があるのです。</p>
<p>今回は、構造化例外処理を活用するためにも、改めて、この Finally ブロックの役割を確認します。</p>

<p>例 1 にあるように、①の Try から④の End Try までが、1 つのまとまった構造化例外処理になります。その内部は、Try ブロック、Catch ブロック、および Finally ブロックの 3 種類のブロックに分かれていています。用語として注意すべき点は、一般に「ブロック」というと、ブロック開始のキーワードから、それと対となる End で始まるキーワードまでを指しますが、①の Try から④の End Try までの全体を Try ブロックとは呼ばない点です。上記の例では、Try ブロックは①から②の
 Catch の直前までです。そして、②から③の直前までが Catch ブロックとなり、③以下の最後のブロックが Finally ブロックです。</p>
<p>Try ブロックの中には、例外が発生する可能性のあるコードを記述します。例外が発生しない場合、そのまま Try ブロックの中が 1 行ずつ実行されます。そして例外が発生せずに Try ブロックの実行が正常に終了すれば、最後に Finally ブロックが実行されます。</p>
<p>もし、Try ブロックの中で例外が発生した場合は、その時点で Catch ブロックに制御が移ります。実際には、Catch ブロックは 1 つだけでなく、例外の種類などに応じて連続的に複数個の Catch ブロックを記述することもできますが、ここでは Finally ブロックに焦点を当てているので、今回はその話は割愛します。そして、Catch ブロックを実行した際も、最後に Finally ブロックを実行します。</p>
<p>つまり、Finally ブロックは、Try ブロックがすべて実行され正常に終了した場合と、Try ブロックの途中で Catch ブロックに制御が移動して例外処理が実行された場合の、両方の場合において、その後に続く「共通の処理」を書くべき場所です。</p>
<p style="margin-top:20px"><a href="#top"><img src="16796-image.png" alt=""> ページのトップへ</a></p>

<h2 id="03">3. Finally ブロックの基本的な役割</h2>
<p>Finally ブロックで行う共通の処理とは、具体的には、終了処理 (後処理) と考えるのが自然です。</p>
<p>例えば、まず一般に何らかの処理を行う場合、準備として何らかの初期化を行うことがあります。例 1 では Try ブロックに書かれた Prepare メソッド呼び出しが初期化と考えてください。そして、続く ProcA メソッドと ProcB メソッドの呼び出しが目的となる処理です。さらに、一般に初期化があれば、それと対になる「後片付け」として後処理が必要です。例えば、ファイルのオープン操作があれば、それと対になってクローズ操作があります。例 1 では、Finish() メソッド呼び出しが後処理と考えてください。この後処理は、Try
 ブロックをすべて正常に実行した場合も必要ですし、例外が発生した場合も一般に後片付けとして終了処理は必要です。このような共通の後処理を行うことが、Finally ブロックの主な役割です。</p>
<p>しかし、次のような疑問を持たれるかも知れません。正常時と例外発生時の「共通の処理」というのであれば、Finally ブロックを使用せずに、次の例 2 や例 3 でもよいのではないでしょうか。</p>

<p>また、例 3 の場合では、Try ブロックの中の①と Catch ブロックの中の②に、Finish メソッド呼び出しがあります。呼び出しは 2 か所あり、Try ブロックと Catch ブロックの両方から、後処理の Finish メソッドを共通で使用しています。</p>

<h2 id="04">4. Finally は Try ... Catch ... Finally を途中で抜けても必ず実行される</h2>
<p>ループやメソッドなど、様々なブロック構造の処理を記述するとき、状況によっては、ブロックの途中でブロックを抜けるなどして、ブロックのそれ以降の処理をスキップしたい場合があります。Visual Basic では、そのような状況に対応できるように Exit 文や Return 文などがあります。例えば、次の例 4 の①ようにメソッドの途中に Return 文を記述すれば、①の If 文の条件を満たす場合は Return が実行され、このメソッドを抜け出して呼び出し元に戻るので、メソッド内のそれ以降の処理 (ProcC
 メソッドの呼び出し) をスキップできます。(この Return 文は、Exit Sub 文と同じです。) このような記述を使用することで、ブロック構造の中で柔軟な実行制御を表現できます。</p>
<p><strong>例 4. Return による処理のスキップ (処理の中断)</strong></p>
<div class="CodeHighlighter">
<div style="clear:both">
<div class="scriptcode">

<div class="title">Visual Basic</div>

<span class="hidden">vb</span>
<pre class="hidden"><code class="csharp">Private Sub Proc1()
    ProcA()
    If FuncB() &gt; 10 Then Return	&larr;①
    ProcC()
End Sub</code></pre>
<pre id="codePreview" class="vb"><code class="csharp">Private Sub Proc1()

同様に、Catch ブロック内の②にも Return 文があり、Func2 関数の戻り値が True の場合は、この例外処理では PrintLogB メソッドをスキップすることになります。しかし、③の Finally ブロックは必ず実行されます。</p>
<p>このように、Return 文などで Try ... Catch ... Finally 構造から強制的に抜け出ても、必ず Finally ブロックは実行されるのです。この仕組みは言語仕様として定義されており、前述の例 2 や例 3 では実現できません。例 2 や例 3 では、ブロックの途中で Return 文を実行すると、それ以降に記述された Finish メソッド呼び出しまでもスキップしてしまいます。</p>

<h2 id="05">5. Finally は入れ子の構造化例外処理にも対応</h2>
<p>もう 1 つ、典型的な例を確認しましょう。それは、次のように実質的に例外処理の入れ子が構成されているケースです。(このような実装は、実際の開発でも頻繁におこなわれることでしょう。)</p>
<p><strong>例 7. 実質的な例外処理の入れ子</strong></p>
<div class="CodeHighlighter">
<div style="clear:both">
<div class="scriptcode">
<div class="pluginEditHolder" pluginCommand="mceScriptCode">
<div class="title">Visual Basic</div>
<div class="pluginEditHolderLink">Edit Script</div>
<span class="hidden">vb</span>
<pre class="hidden"><code class="csharp">Private Sub ParentProc()
    Try				&larr;①
        ChildProc()		&larr;②
    Catch ex As Exception	&larr;③
        '例外処理
    End Try
End Sub

Private Sub ChildProc()
    Try				&larr;④
        Prepare()
        ProcA()			&larr;⑤
        ProcB()
    Catch ex As FileIOException	&larr;⑥
        '例外処理
    Finally
        Finish()				&larr;⑦
    End Try
End Sub</code></pre>
<pre id="codePreview" class="vb"><code class="csharp">Private Sub ParentProc()

        ChildProc()		&larr;②
    Catch ex As Exception	&larr;③
        '例外処理
    End Try
End Sub

Private Sub ChildProc()
    Try				&larr;④
        Prepare()
        ProcA()			&larr;⑤
        ProcB()
    Catch ex As FileIOException	&larr;⑥
        '例外処理
    Finally
        Finish()				&larr;⑦
    End Try
End Sub</code></pre>
</div>
</div>

<p>この例では、冒頭の ParentProc メソッドの中に、①の構造化例外処理があります。そして、①の Try ブロックの中の②では、ChildProc メソッドを呼び出しており、その ChildProc メソッドの中にも、④のように構造化例外処理があります。これは実質的には入れ子と見なすことができ、①の Try ブロックの中に、ChildProc メソッドの中の構造化例外処理 (④) があると考えることができます。</p>
<p>このとき、入れ子の内側の Try ブロックに記述された⑤の ProcA メソッドで SqlException という種類の例外が発生したとします。このとき、まず⑥の Catch ブロックが例外を処理できるか確認されますが、⑥は「As FileIOException」と記述しているので「SqlException」という例外を処理できません。よって、さらに入れ子の外側で処理可能な Catch ブロックを探すことになります。すると、呼び出し元の ParentProc メソッドまで戻り、③に「As Exception」と記述された汎用的な例外処理を行える
 Catch ブロックに制御が移ります。</p>
<p>このように、Try ブロックに直接対応する Catch ブロックで処理が出来ない場合は、入れ子構造の外側の Catch ブロックに処理できるブロックがないか確認され制御が移ります。これは、例外処理を実装するときの典型的なパターンです。</p>
<p>このときも、⑤で例外が発生した際に、Finally ブロックの⑦の後処理のメソッドは確実に実行されます。これも Finally ブロックだから出来るのであって、例 2 や例 3 では後処理を確実に実行できません。</p>
<p>また、前述の ChildProc メソッドで直接例外を処理せずに、単に呼び出し元に例外を振る実装も考えられるでしょう。その場合、コードは次のようになります。</p>

<p>この例では、Try ブロックと Finally ブロックしかありません。これも文法的に正しい構文です。この場合、例外が発生するとその例外への対処はこの ChildProc メソッドでは行わず、呼び出し元の Catch ブロックに振ることになり、例外の発生は直ぐに呼び出し元に制御が移ることを意味します。しかし、その際でも、この Finally ブロックは確実に実行されます。</p>
<p>この例などは、例外に対処する処理がないので、まさに「後処理を確実にするために Finally ブロックがある」という、その存在意義が感じられるでしょう。もちろん、この例の Try ブロックの中で、Return 文を実行して呼び出し元に戻るとしても、Finally ブロックは確実に実行されます。</p>
<p><br>
今回は、Finally ブロックについて、その役割を確認しました。構造化例外処理の一部として Finally ブロックは扱われていますが、上記のように、Catch ブロック無しでも利用でき、確実に後処理を実行するため手段として利用できます。いわば、Finally ブロックは、「構造化後処理」を実現するための手段と言ってもよいでしょう。</p>
